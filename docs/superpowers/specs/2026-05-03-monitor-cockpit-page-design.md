# AI 运行监控驾驶舱页面设计

**日期**：2026-05-03

## 目标

为当前 `web-console` 新增一个独立的 AI 运行监控驾驶舱页面，而不是继续把所有监控信息堆在首页卡片中。

该页面的目标有三点：

- 让运行健康信息具备“单独成页”的展示力度，适合作为平台核心能力页面与论文截图页面。
- 用圆形百分比、服务矩阵、任务流转和告警列表，替代当前首页中信息稀薄、可读性弱的监控卡片。
- 把“没有得到答案”从模糊兜底文案升级为“缺失原因解释”，明确告诉用户为什么当前没有某项监控数据。

## 已确认结论

- 用户明确接受“新开一个界面”的方向，而不是只在 `DashboardView` 上继续堆内容。
- 页面视觉风格采用偏“状态驾驶舱”的方案，而不是偏白卡片的普通后台方案。
- 页面信息密度需要明显高于当前首页监控卡片，不能只显示 4 到 5 个数字。
- 页面必须保留真实数据边界，不能为了好看伪造后端没有返回的监控字段。
- 对缺失数据的处理原则是“解释为什么缺失”，而不是继续留白或只显示“暂无”。

## 当前问题

当前首页监控面板存在以下问题：

- 页面层级太浅，主要仍是几块白卡片，视觉上难以体现“实时监控”。
- 当前只把 `CPU` 百分比直接显示出来，其余关键区域没有形成统一的数据表达方式。
- `任务上下文`、`关键服务`、`加速器` 等模块在数据缺失时只显示简单兜底文案，用户无法判断是页面问题、服务问题还是当前没有任务。
- 服务数量、任务上下文、告警等信息过于压缩，缺少展开后的具体内容。
- 首页同时承担平台概览与监控聚合，信息角色冲突，导致监控面板看起来像“附属模块”而不是“核心能力页面”。

## 设计决策

采用“首页保留简版监控摘要 + 新增独立监控驾驶舱页面”的方案。

- 首页 `DashboardView` 继续保留一块简版监控摘要，承担入口和状态预览作用。
- 新增一个独立页面，例如 `MonitorCockpitView.vue`，作为监控驾驶舱主页面。
- 左侧导航新增“运行监控”入口，与“平台总览 / 生成工作台 / 历史中心 / 模型中心 / 任务审计”并列。
- 驾驶舱页面集中展示更高信息密度的监控内容，避免继续让首页承担详细运行分析职责。

这样处理的原因：

- 首页职责回到“平台概览”。
- 监控能力具备单独叙事空间，视觉和内容都更完整。
- 后续如果监控服务继续扩展字段，这个页面比首页更容易承接扩展。

## 页面信息架构

新页面采用自上而下的五层结构。

### 1. 顶部状态总览区

用于一眼说明当前平台是不是健康、是不是正在实时连接，以及当前采样属于什么环境。

建议展示：

- 页面主标题：`AI 运行监控驾驶舱`
- 副标题：强调实时监控、AI 运行健康、服务状态联动
- 实时连接状态：`实时连接中 / 连接已断开`
- 当前平台标签：`macOS / Windows`
- 采样时间标签：展示最后一次快照时间
- 总体健康等级：`healthy / warning / critical`

### 2. 关键摘要指标区

用于把最重要的监控结论先抛给用户。

建议展示 4 个摘要块：

- 总体健康
- 活跃任务数
- 关键服务在线数
- 活跃告警数

每个块不仅展示数字，还要展示一行简短解释，例如：

- `统一内存压力偏高，但链路可继续运行`
- `最近阶段为 scoring`
- `1 个服务尚未上报快照`
- `当前包含 2 条活跃告警`

### 3. 资源圆环与资源详情区

这是页面的核心视觉区，解决“需要有一个圆圈来表示百分比”的需求。

采用三个主圆环：

- `CPU`
- `内存`
- `GPU / MPS`

每个圆环的规则：

- 有真实百分比时，显示圆环和中心数值。
- 没有真实百分比时，不伪造数值，改为 `N/A` 或 `--`，并在说明文字里写缺失原因。
- 颜色与健康等级对应：
  - 绿色：健康
  - 黄色或橙色：警告
  - 红色：严重

资源详情区补充两块说明：

- 宿主机摘要
  - 平台类型
  - CPU 使用率
  - 可选的内存压力、磁盘状态、交换区状态
- 加速器详情
  - `apple-mps / nvidia-cuda / unavailable`
  - GPU 名称或 MPS 可用性
  - VRAM 或统一内存相关说明
  - 不可用时展示 `unavailable_reason`

### 4. 缺失原因与告警解释区

这是为了解决“有些是没有得到答案是为什么”。

该区域不是简单说“暂无”，而是明确写成“为什么缺失”。

至少覆盖以下三类：

- 任务上下文缺失
  - 当前没有运行任务，所以 `latest_task_stage` 没有上报
  - 或任务已提交但阶段回调尚未到达
- 关键服务快照缺失
  - `service_snapshots` 当前为空
  - 或部分服务尚未完成探针注册
- 资源指标缺失
  - 后端未返回 `memory_*`
  - 未返回 `gpu_utilization_percent`
  - 当前平台无法稳定采集该指标

这一块应当显式告诉用户：

- 这是“没有数据”
- 不是“前端没有渲染”
- 不是“页面坏了”

### 5. 服务矩阵与任务流转区

这一层负责把被压缩掉的细节摊开。

服务矩阵建议展示：

- 服务名称
- 当前状态
- 可选的运行说明
- 可选的进程内存占用

任务流转建议展示：

- 当前活跃任务条数
- 每条任务的最新阶段
- 对阶段的解释性文案
- 如果没有活跃任务，则显示“当前为什么没有任务上下文”

## 页面与现有首页的关系

首页保留轻量监控摘要，不再承载完整监控细节。

首页建议保留：

- 当前平台
- CPU 简略状态
- 总体健康
- 活跃告警数
- 一个“进入监控驾驶舱”的入口按钮

首页不再负责：

- 详细服务矩阵
- 详细告警说明
- 多个圆环资源图
- 详细任务流转

## 路由与导航设计

需要新增一个独立路由，例如：

- `/monitor`

需要同步更新：

- `web-console/src/router/index.ts`
- `web-console/src/components/AppShell.vue`

导航项建议：

- 标签：`运行监控`
- 英文提示：`Monitor`

页面标题建议：

- 顶栏标题：`AI 运行监控驾驶舱`

## 数据映射设计

页面必须建立“真实字段 -> 可视模块”的明确映射。

### 来自 `MonitorOverview` 的直接字段

- `overall_health`
  - 映射到总体健康摘要、状态标签、颜色体系
- `host_snapshot.platform_family`
  - 映射到平台标签
- `host_snapshot.cpu_usage_percent`
  - 映射到 CPU 圆环
- `host_snapshot.memory_total_bytes`
  - 与 `memory_used_bytes` 一起计算内存百分比
- `host_snapshot.memory_used_bytes`
  - 计算内存百分比
- `host_snapshot.memory_pressure_level`
  - 映射到宿主机说明文案
- `host_snapshot.swap_used_bytes`
  - 可映射到宿主机补充说明
- `accelerator_snapshot.accelerator_type`
  - 映射到 GPU/MPS 标题和文案
- `accelerator_snapshot.gpu_name`
  - 映射到加速器详情
- `accelerator_snapshot.gpu_utilization_percent`
  - 映射到 GPU 圆环
- `accelerator_snapshot.vram_total_mb`
  - 与 `vram_used_mb` 一起计算 VRAM 百分比
- `accelerator_snapshot.vram_used_mb`
  - 计算 VRAM 百分比
- `accelerator_snapshot.available`
  - 映射到可用状态标签
- `accelerator_snapshot.unavailable_reason`
  - 映射到缺失原因面板
- `accelerator_snapshot.unified_memory_pressure`
  - 用于 Apple MPS 说明文案
- `service_snapshots`
  - 映射到关键服务在线数与服务矩阵
- `task_runtime_context.active_task_count`
  - 映射到活跃任务数
- `task_runtime_context.latest_task_stage`
  - 映射到最新阶段与任务流转说明
- `active_alerts`
  - 映射到活跃告警数与告警列表
- `recent_alerts`
  - 可映射到最近告警区域

### 计算字段

页面需要在前端计算以下派生值：

- 内存使用率
- VRAM 使用率
- 在线服务数 / 总服务数
- 总体颜色主题
- 是否存在缺失原因
- 缺失原因的优先级顺序

## 缺失原因策略

这是本设计最重要的行为约束之一。

规则如下：

- 只要字段缺失，就必须优先尝试解释缺失原因。
- 如果能从已有字段推断原因，则展示推断解释。
- 如果无法安全推断，就展示“监控服务暂未提供该字段”。
- 不允许把缺失数值伪装成 `0%`。

示例：

- `active_task_count = 0` 且 `latest_task_stage` 为空
  - 展示：`当前没有运行任务，因此没有最新阶段上报`
- `service_snapshots.length = 0`
  - 展示：`尚未收到关键服务快照，说明服务探针尚未返回`
- `memory_total_bytes` 或 `memory_used_bytes` 缺失
  - 展示：`监控服务尚未提供内存占用字段`
- `accelerator_type = unavailable`
  - 展示：`当前设备未启用可用 AI 加速器`

## 视觉语言

页面应保持驾驶舱方向，但仍然服务于论文与教学展示。

视觉要求：

- 深色背景为主，带轻微蓝色或青色工业氛围
- 圆环百分比为第一视觉焦点
- 信息卡片采用分层玻璃或金属面板感
- 保证亮色文本对比度，不做花哨但难读的装饰
- 比首页更强烈，但不能做成泛科技大屏风格

## 响应式要求

- 桌面端优先呈现完整驾驶舱布局
- 窄屏下摘要区、圆环区、服务矩阵区按纵向堆叠
- 圆环在窄屏下仍需要保持中心数值清晰
- 服务矩阵在移动端允许压缩为两列信息卡，而不是强行保留表格宽度

## 实现约束

- 复用现有 `platformStore` 中的 `monitorOverview` 与 `monitorConnected`
- 不新增新的监控接口协议
- 首版优先在前端重组现有字段与解释逻辑
- 允许新增前端派生工具函数或视图级计算属性
- 不要求首版补齐所有后端指标，只要求把已有字段用得更清楚

## 测试要求

至少覆盖以下前端测试：

- 新路由可正常渲染监控页面
- 有 `CPU` / `GPU` / `内存` 数据时，圆环中心显示正确数值
- 缺失字段时，不显示伪造的 `0%`
- 缺失字段时，能显示对应原因文案
- `service_snapshots` 非空时，服务矩阵显示服务名称与状态
- `service_snapshots` 为空时，显示服务快照缺失原因
- `active_task_count = 0` 且阶段缺失时，显示任务上下文缺失原因

## 推荐实施方式

推荐按以下边界拆分实现：

- 新增页面视图：`MonitorCockpitView.vue`
- 视图辅助函数：负责圆环数值格式化、缺失原因推导、颜色等级映射
- 首页摘要调整：为监控驾驶舱提供入口，而不是继续扩张首页卡片
- 导航调整：在壳层中加入独立入口

## 验证方式

- `npm --prefix web-console run test -- DashboardView.spec.ts`
- `npm --prefix web-console run test -- router/index.spec.ts`
- 新增监控页面测试通过
- `npm --prefix web-console run build`

## 本设计不包含

- 新增后端监控字段
- 改造监控服务协议
- 自动修复告警
- 任务控制能力

本轮只聚焦“新开一个监控界面，并把现有监控信息做得更完整、更直观、更可解释”。
