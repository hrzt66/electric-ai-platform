# YOLO Superclass Rebuild Design

## Goal

Rebuild the auxiliary YOLO detector used by `electric-score-v2` so the final detector is aligned with the user's real prompt distribution and can realistically target `mAP50 ~= 0.7` and `mAP50-95 ~= 0.5` on the merged validation set.

## Problem Summary

The current detector underperforms for three coupled reasons:

1. The effective training run was not consistently using the intended merged dataset.
2. The class taxonomy is too component-fragmented and does not match the user's dominant scene prompts.
3. Several prompt families lack enough labeled data, so the model cannot ground those prompts reliably.

The result is a detector that produces low recall on the bundled validation metrics and weak prompt-grounding for the downstream scoring runtime.

## Success Criteria

- Final auxiliary YOLO validation target:
  - `mAP50 >= 0.70`
  - `mAP50-95 >= 0.50`
- Training and validation must use the same merged dataset definition.
- Prompt-grounding aliases must match the new superclass taxonomy.
- New public scene images for wind, solar, hydro, control-room, and night-maintenance must be downloaded into the project dataset tree with attribution metadata.
- The implementation must remain runnable on this local macOS MPS environment without requiring system-wide toolchain changes.

## Scope

This work covers:

- Rebuilding the YOLO taxonomy into stable superclasses.
- Adding new labeled detection sources for missing domains.
- Adding new public scene images for generation/scoring support.
- Unifying merge, split, training, validation, and reporting code paths.
- Updating prompt aliasing and detector-sensitive scoring logic.

This work does not cover:

- Replacing the full scoring model architecture.
- Replacing the frontend score display model.
- Training a new control-room-specific detector unless a high-quality labeled source is found.

## Recommended Taxonomy

The detector will use the following 7 YOLO superclasses:

1. `substation_primary`
2. `transmission_tower`
3. `insulator_string`
4. `wind_turbine`
5. `solar_panel`
6. `dam`
7. `maintenance_ppe`

`control_room` will not be a YOLO superclass in this rebuild. It will remain a scoring-only scene concept handled through prompt/image heuristics because no reliable, project-appropriate real-scene detection dataset has yet been identified. Forcing it into YOLO would likely reduce the overall validation metrics.

## Data Strategy

### Strong detection sources

The merged detector dataset should be built from publicly available labeled sources that map cleanly into the 7 superclasses:

- Existing power equipment detection data already in the repo:
  - substation object detection
  - transmission line object detection
  - powerline components and faults
- New public detection sources:
  - BAH24 object detection dataset for substation, tower, solar, wind, and check-dam
  - TTPLA for tower and line scene support
  - RSIn for insulator support
  - Wind Turbine Detection for wind turbine support
  - Solar Panel Detector for solar panel support
  - MIS Check-Dam dataset for dam support
  - PPE workplace dataset for maintenance PPE support

### Scene-image support data

For generation/scoring support, download additional public scene images into `model/datasets/generation-v3/raw/` for:

- wind
- solar
- hydro
- control-room
- night-maintenance

These scene images do not need YOLO boxes, but they must include attribution sidecars and stay within the project's existing public-download pipeline conventions.

## Mapping Rules

Existing fine-grained component labels will be remapped into superclasses:

- `bus`, `breaker`, `switch`, `arrester`, `bushing`, `ct`, `pt`, `capacitor`, `gis`, `frame`, `filter`, `pipe` -> `substation_primary`
- `tower`, `frame` when used as line structure context -> `transmission_tower`
- `insulator`, `broken insulator` -> `insulator_string`
- `line`, `cable` are not standalone final classes; they remain supporting topology signals and may be retained as internal features if needed, but not as final exported YOLO classes in this rebuild
- wind-related labels -> `wind_turbine`
- solar-related labels -> `solar_panel`
- dam/check-dam labels -> `dam`
- helmet/vest/worker PPE labels used for maintenance scenes -> `maintenance_ppe`

When a source label could map to more than one superclass, the mapping will be deterministic and source-specific.

## Training Architecture

### Detector training

- Use a single merged YOLO dataset artifact generated by the pipeline.
- Use one unified training entrypoint and one unified validation entrypoint.
- Persist the exact merged `dataset.yaml` used for both train and validation.
- Default model target:
  - start with `yolov8s.pt`
  - fall back to `yolov8n.pt` only if memory pressure on MPS makes `yolov8s` unstable
- Default training profile target:
  - `imgsz=640`
  - conservative batch size tuned for local MPS memory
  - moderate augmentation, not the previous ultra-disabled profile
  - validation every epoch
  - save best weights

### Dataset balancing

To reach the target metrics, the merged dataset must not stay dominated by insulator/line examples. The rebuild will:

- drop unsupported or tiny classes from YOLO export
- rebalance source contributions where one source overwhelms the merge
- enforce per-class minimum instance thresholds
- record per-class train/val counts in the training report

## Runtime and Scoring Changes

- Update `PROMPT_CLASS_ALIASES` to the new 7-class schema.
- Update prompt-grounding logic so text-consistency and physical-plausibility scores rely on the new superclasses.
- Keep non-YOLO scene concepts like `control_room` in the explanation layer without forcing them into the detector.

## Verification Plan

The rebuild is complete only when all of the following are true:

- new public images exist on disk for wind, solar, hydro, control-room, and night-maintenance
- merged YOLO dataset contains the rebuilt 7 classes
- training report and bundled metrics point to the same merged dataset path
- final validation reaches or closely approaches the target band
- runtime bundle loads with the new class list
- targeted tests for dataset remapping, aliasing, and merge behavior pass

## Risks and Mitigations

- Risk: control-room data remains weak
  - Mitigation: keep it outside YOLO and handle it in scoring logic only
- Risk: MPS memory pressure during larger-model training
  - Mitigation: tune batch size first, then model size, without changing the target taxonomy
- Risk: downloaded public sources differ in label quality
  - Mitigation: source-specific remapping and class filtering before merge
- Risk: metric target is still missed after naive merge
  - Mitigation: iterate on balancing and class pruning before changing architecture
