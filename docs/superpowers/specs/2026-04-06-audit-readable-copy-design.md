# Audit Readable Copy Design

**日期**：2026-04-06

## 目标

将前端审计展示从“英文事件名 + 原始 JSON”改为“中文标题 + 中文说明”，让用户在生成工作台、任务审计页和历史详情抽屉中都能直接读懂任务执行过程。

## 方案对比

### 方案 A：只做前端格式化函数

- 新增一个函数，把审计事件转换成中文文案
- 三个页面继续各自维护时间线模板

优点：
- 改动最小
- 风险低

缺点：
- 三处模板重复
- 后续样式和空态文案容易再次分叉

### 方案 B：共用格式化器 + 共用审计时间线组件

- 新增审计格式化器，负责标题翻译与 payload 摘要
- 新增共用时间线组件，统一渲染空态、标题、正文
- 生成页、任务审计页、历史详情抽屉全部接入共用组件

优点：
- 三个入口展示一致
- 后续新增事件类型时只需要补一处映射
- 不再出现有的页面修好了、有的页面仍显示 JSON 的情况

缺点：
- 比方案 A 多一个组件抽取步骤

### 方案 C：后端直接返回中文展示文案

- 审计服务或 Python 运行时直接生成中文字符串

优点：
- 前端逻辑更少

缺点：
- 展示层语言写死在后端
- 会把本来结构化的数据接口变成文案接口
- 后续多语言、样式重构、前端分场景展示都受限

## 采用方案

采用 **方案 B：共用格式化器 + 共用审计时间线组件**。

原因：
- 当前问题不只是“内容太难读”，也是“三处页面重复直接渲染后端字段”的结构问题。
- 抽出共用组件后，可以在不修改后端 API 的前提下，稳定统一所有审计展示。

## 设计说明

### 审计格式化规则

前端新增一个审计事件展示器，对每条事件生成：

- `title`：中文阶段标题
- `description`：适合阅读的中文说明

优先覆盖当前真实运行链路中的核心事件：

- `task.preparing`
- `model.prepare`
- `generation.completed`
- `task.completed`
- `task.failed`

同时兼容旧事件或宽泛事件名：

- `job_generating`
- `generating`
- `scoring`
- `persisting`
- `queued`
- `completed`
- `failed`

对于未知事件：

- 不再展示原始 JSON
- 尝试把 payload 解析成“字段名 + 字段值”的中文短句
- 若无 payload，则回退到 `message`
- 若二者都没有，则显示“该阶段已记录，但没有附加说明”

### 共用审计时间线组件

新增一个共用组件接收：

- `events`
- `emptyDescription`

组件内部完成：

- 空态渲染
- 审计事件中文标题与中文说明渲染
- 时间线样式统一

### 页面接入范围

以下三个入口统一改造：

- 生成工作台右侧“实时状态”
- 任务审计页“时间线”
- 历史详情抽屉“任务审计轨迹”

## 不做的内容

- 不修改后端审计接口结构
- 不新增数据库字段
- 不把原始 payload 删除，仍保留在接口中供后续诊断使用

## 验证方式

- 单测：审计事件格式化器的核心事件中文化输出
- 前端全量测试：`npm --prefix web-console run test`
- 前端构建：`npm --prefix web-console run build`
