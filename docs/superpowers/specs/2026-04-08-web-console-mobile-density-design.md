# Web Console 手机端薄卡片化设计

## 背景

前一轮移动端改造已经完成了功能层面的响应式适配，桌面端结构也基本保持稳定，但手机端观感仍然偏厚重。当前问题不是“不能用”，而是“太臃肿”：顶部栏和底部导航占高偏大，生成页与历史页堆叠了过多高卡片，卡片留白、圆角和阴影都偏重，导致首屏有效信息过少。

本轮设计不再扩大移动端功能范围，也不重做桌面端视觉语言，而是聚焦于一个更明确的目标：**让手机端看起来更像轻量原生应用，而不是把桌面端大面板压缩到小屏里。**

## 目标

1. 在不改变桌面端整体视觉和信息结构的前提下，显著降低手机端页面的厚重感。
2. 保留卡片语义，但让手机端卡片高度接近列表项，而不是大面板。
3. 优先提升手机端首屏有效内容密度，减少一屏内无效留白和重复文案。
4. 保持全部现有功能入口可达，不通过删除功能来换取“简洁”。

## 非目标

1. 不新增路由、业务流程或后端接口。
2. 不重做整站品牌视觉，不改桌面端主要布局。
3. 不将手机端整体切换为完全无卡片的纯列表体系。
4. 不引入新的全局状态管理结构。

## 设计结论

本轮采用用户确认后的方向：

**以 A“薄卡片系统”为主，吸收少量 B“列表条目系统”的做法。**

也就是说，手机端仍然保留卡片边界和模块感，但：

1. 卡片在视觉上更薄、更扁、更轻。
2. 大部分卡片在手机端的高度要接近列表项高度。
3. 次要说明文、重复副标题、冗长摘要默认收缩或隐藏。
4. 结构上优先让页面像“连续的信息流”，而不是“高卡片堆叠墙”。

## 全局设计原则

### 1. 薄卡片规范

手机端统一采用“薄卡片”规范：

1. `padding` 明显缩小，优先压缩上下留白。
2. 圆角从桌面端的大圆角收紧为中小圆角。
3. 阴影从强投影改为浅投影，减少厚重感。
4. 卡片内部标题、meta、操作尽量在两行内解决。
5. 超过两行的说明文默认不直接出现在手机端主流中。

### 2. 信息优先级收束

手机端每个模块只保留一个主要阅读点：

1. 只保留一个主标题。
2. 副标题、英文提示、辅助 badge 默认弱化、隐藏或并入一行 meta。
3. 同一块内容里不重复出现“标题 + 解释 + 再解释”的层级。
4. 所有可延后阅读的信息都后移到抽屉、折叠区或详情层。

### 3. 手机端高度优先

本轮优先优化“垂直高度”而非“横向排布”：

1. 顶部栏更薄。
2. 底部导航更薄。
3. 中间内容区卡片更薄。
4. 首屏只保留最重要的信息和主操作。

### 4. 桌面端隔离

所有本轮视觉密度改动优先限制在 `<= 768px` 的手机断点内。桌面端只允许接收极小的兼容性修正，不做审美层级重排。

## 页面设计

### AppShell

手机端外壳调整为“轻顶栏 + 薄底部导航”：

1. 顶部栏只保留菜单按钮、当前页面标题、极小状态点或极简状态标签。
2. 现有长 badge 文案在手机端移除，避免占据整行高度。
3. 底部导航保留 5 个主入口，但高度缩小，英文 hint 弱化到可隐藏或极低权重。
4. 当前态仍要足够清晰，但避免使用大面积高亮块。
5. 主内容区为底部导航预留空间时，也同步减少额外外边距，避免出现双重空白。

### LoginView

登录页在手机端从“工业舞台 + 登录大卡”调整为“轻介绍 + 薄登录卡”：

1. 首屏优先显示平台标题、登录表单和主按钮。
2. 工业说明区保留，但文案明显减量，不在首屏堆大段介绍。
3. 状态卡和 system board 下沉到下方，并改为薄条式说明块。
4. 默认账号提示保留，但压缩成简短提示行，不再占据过高区域。
5. 表单输入区和按钮保留足够可点面积，但去除不必要的大留白。

### GenerateView

生成页是本轮最优先处理的页面。

手机端采用“结果优先 + 短条模块”的表达：

1. 结果预览继续放在最上方。
2. 预览区头部压薄，只保留必要标题、状态和关键 meta。
3. 缩略图区继续可滑动，但整体边框、padding 和留白全部缩小。
4. 参数入口改成短条按钮或轻量入口，不再像独立大卡。
5. 进度卡改为薄型状态条块，只保留百分比、当前阶段、少量阶段概览。
6. 评分区优先显示总分和少量关键指标；雷达图如保留，也要降低占高，避免成为独立高块。
7. 审计区默认只露出最近一到两条，不直接展开成长时间轴大卡。
8. 生成页在手机端形成连续短模块流，不再出现“结果卡 + 大按钮卡 + 大进度卡 + 大评分卡 + 大审计卡”的连续高堆叠。

### ParameterPanel

参数面板在手机端继续走抽屉路线，但抽屉内部也需要薄化：

1. 表单项间距缩小。
2. 模型说明卡弱化，冗长描述只保留一行到两行。
3. 数值输入、滑块和文本框布局更紧凑。
4. 提交按钮保留主按钮识别度，但不再使用厚重的大按钮外观。

### ResultPreview

结果预览在手机端保留主图优先，但减少配套装饰：

1. 主图区域略缩短，避免首屏被单张图过度占满。
2. meta 信息合并到更短的一行或两行。
3. 缩略图尺寸缩小，边框弱化。
4. 标题区与标签区不再占据明显额外高度。

### GenerationProgressCard

进度展示从“独立面板”向“轻状态块”收束：

1. 阶段摘要保留，但阶段卡变薄。
2. 最近事件条目减少，只保留最关键的两条左右。
3. 进度说明文改短，避免块内多层文字堆积。

### ScoreRadar

评分区在手机端不再强调大面积图形：

1. 总分优先。
2. 四维指标用短条或短列表表达。
3. 雷达图若继续保留，则缩小比例并作为补充信息，而非主体。
4. 手机端不得再出现“图 + 长列表”双重占高。

### HistoryView / HistoryTable / HistoryFilters / HistoryDetailDrawer

历史页是第二优先页面。

1. 筛选区默认收起，展开后也使用紧凑表单间距。
2. 历史列表从“大卡片”改成“短横卡”：
   - 左侧小缩略图
   - 右侧一行标题
   - 一行 meta
   - 提示词摘要缩到极短，必要时直接隐藏
3. 每条记录默认高度尽量接近列表项，而不是详情卡。
4. 历史详情抽屉仍保留完整信息，但正文内部也采用更紧凑的间距和更薄的评分块。

### Dashboard / ModelCenter / TaskAudit

这三个页面不单独重构结构，只统一套用手机端薄卡片规范：

1. 所有摘要卡降低高度。
2. 长说明文默认缩短或隐藏。
3. 列表、标签和状态块尽量单行化。
4. 优先提高首屏可见内容数量，而不是保留桌面端的展示气质。

## 交互与行为

1. 本轮不改变已有主流程行为，主要调整视觉密度和移动端的信息展开顺序。
2. 折叠、抽屉、详情层继续保留，但其目标变成“承接次要信息”，而不是解决布局失败。
3. 页面首屏必须在用户不额外展开的情况下，直接显示最关键结果和主操作。

## 验收标准

满足以下条件即可认为本轮设计达到目标：

1. 手机端首屏可见的有效信息明显多于前一版。
2. 手机端页面不再出现连续多个高大白卡片堆叠的观感。
3. 手机端顶部栏和底部导航明显更薄，不抢占内容区。
4. 生成页和历史页在手机端的观感从“厚面板”转为“短模块流”。
5. 桌面端视觉结构不出现明显回归。

## 实施顺序建议

1. 先改 `AppShell.vue`，统一收紧手机端壳层高度。
2. 再改 `GenerateView.vue` 及其关联组件，这是感知变化最大的页面。
3. 然后改 `HistoryView.vue` 及历史相关组件，统一把大卡片收成短横卡。
4. 最后补齐 `LoginView.vue`、`DashboardView.vue`、`ModelCenterView.vue`、`TaskAuditView.vue` 的薄卡片收口。
