# Web Console 全站手机端适配设计

## 背景

当前 `web-console` 已形成完整的桌面端控制台体验，包含登录、概览、生成工作台、历史中心、模型中心与任务审计等页面。整体风格统一，但核心布局依赖侧边栏、多列卡片和桌面表格，在手机端会出现信息拥挤、横向溢出与主操作链过长的问题。

本次设计目标是在不删除功能、不重写业务流程的前提下，为整个前端增加稳定的手机端展示能力，并尽量保持桌面端体验不受影响。

## 目标

1. 整个前端在手机端可正常浏览、导航和操作，不依赖页面横向滚动。
2. 保留所有现有功能，只调整入口位置、信息顺序和容器形态。
3. 桌面端现有布局与视觉风格基本保持不变。
4. 为后续实现提供统一的断点策略、布局规则和组件改造边界。

## 非目标

1. 不新增新的业务功能、接口或路由。
2. 不对桌面端进行大规模视觉重做。
3. 不把手机端拆成独立站点或独立路由体系。

## 设计结论

采用 “A 方案为主，吸收 B 方案的底部导航”：

1. 手机端以抽屉、底部导航、纵向滚动和卡片化布局为核心。
2. 现有左侧边栏在手机端收纳为顶部菜单抽屉。
3. 手机端新增底部主导航承接主页面切换。
4. 生成页、历史页和详情层优先改造成更适合手机阅读与触达的单列流式结构。

该方案能保留全部功能，同时最大限度复用现有页面、路由与组件结构，改造风险最低。

## 断点策略

统一使用三层响应式断点：

1. `<= 1100px`
进入窄屏/平板布局。保留主内容结构，但开始减少横向列数，允许摘要和操作区上下重排。

2. `<= 768px`
进入手机主布局。启用底部导航、顶部菜单入口、抽屉化次级信息区、单列内容主流。

3. `<= 480px`
进入小屏优化层。进一步压缩字号、间距、按钮高度和卡片内边距，确保单手设备上的有效可视区域。

## 架构与组件边界

### 公共框架

由 `AppShell.vue` 统一承接手机端全局框架行为：

1. 桌面端继续使用侧边栏 + 顶部栏。
2. 手机端切换为顶部栏 + 底部主导航 + 菜单抽屉。
3. 顶部栏负责显示当前页面标题、状态徽标和菜单入口。
4. 底部导航负责概览、生成、历史、模型、审计五个主页面切换。
5. 抽屉中保留品牌信息、完整导航、当前账号和退出登录入口。

这样可以避免每个页面单独实现导航适配，保证全站导航逻辑一致。

### 页面容器

页面内部优先使用以下原则：

1. 多列转单列或双列转单列。
2. 高频信息放首屏，低频信息收进抽屉或折叠区。
3. 长文本默认支持换行、截断或多行省略。
4. 避免页面级横向滚动，只允许缩略图带、局部列表等小范围横向滚动。

## 页面设计

### 登录页

保留当前工业控制风格，但改为可堆叠的单列布局：

1. 在 `<= 1120px` 时切换为上下结构。
2. 平台介绍、状态卡和系统板块在上方顺序堆叠。
3. 登录卡片占据下方完整宽度。
4. `<= 768px` 时状态卡统一改为单列，标题字号、间距和表单卡片圆角适度压缩。

### 概览页

1. `hero` 区在手机端改为纵向排列，按钮独占一行。
2. 摘要卡从四列降为两列，再降为单列。
3. 最近任务与模型概览改为单列面板，保持卡片式内容组织。

### 生成页

生成页是手机端的核心页面，采用 “结果优先” 顺序：

1. 首屏默认展示结果预览。
2. 结果预览之后展示任务进度与状态。
3. 评分区紧跟其后，改为单列或双列小卡片。
4. 参数面板不再常驻占列，改为按钮触发的底部抽屉或全屏抽屉。
5. 审计轨迹放在后续区域，通过纵向滚动查看。

组件级行为：

1. `ResultPreview.vue`
保留大图预览，缩略图继续横向滑动；元信息允许换行。

2. `ParameterPanel.vue`
桌面端继续作为常驻面板；手机端切换为抽屉内容，表单项全部单列排列。

3. `ScoreRadar.vue` 与 `GenerationProgressCard.vue`
手机端从侧列区域改为结果下方堆叠展示。

4. 生成页整体不再依赖 `height: calc(100vh - ...)` 的固定高度策略，手机端转为自然页面滚动，避免软键盘和浏览器工具栏导致的可视区异常。

### 历史页

历史页在手机端由 “表格” 转为 “卡片列表”：

1. `HistoryFilters.vue`
保留现有筛选项，但手机端改为单列表单，并支持折叠/展开。

2. `HistoryTable.vue`
桌面端继续使用 `el-table`。
手机端切换为卡片列表，每张卡展示：
   - 缩略图
   - 图片名称
   - 提示词摘要
   - 模型
   - 总分
   - 状态
   - 时间
   - 查看详情入口

3. 历史列表的点击行为保持不变，仍然进入详情层。

### 历史详情层

`HistoryDetailDrawer.vue` 的手机端行为调整为：

1. 使用接近全屏的抽屉尺寸。
2. 标题区域固定在上方，内容区域独立滚动。
3. 图片、提示词、采样参数、评分、审计轨迹按阅读顺序单列排布。
4. 评分区从五列改为两列，在更小屏幕下再改为单列。

### 模型中心

1. 模型卡片在手机端统一改为单列。
2. 卡片内部信息保持现有字段，但长路径与提示词支持断行。
3. 顶部摘要区从横向对齐改为纵向堆叠。

### 任务审计页

1. 摘要卡片在手机端改为上下布局。
2. 审计时间线和关联资产从双列改为单列。
3. 关联资产列表保留卡片式展示。

## 数据流与状态

本次适配不引入新的后端依赖，也不改变 Pinia store 结构。

数据流保持现状：

1. 路由仍驱动主页面切换。
2. `usePlatformStore` 仍负责任务、模型、历史和审计数据。
3. 手机端新增的抽屉、折叠、底部导航仅改变展示方式，不改变数据来源。

额外新增的前端状态主要为：

1. 菜单抽屉是否打开。
2. 参数抽屉是否打开。
3. 历史筛选区域是否折叠。
4. 当前是否处于手机布局所需的样式分支。

这些状态应尽量保持局部化，避免污染现有 store。

## 错误处理与兼容性

### 错误处理

现有页面中的告警、空态和加载骨架继续保留，在手机端只调整布局，不改变错误语义。

重点保证：

1. 抽屉打开时仍能显示加载、错误和空态。
2. 无任务、任务进行中、任务失败、无历史等状态在手机端都可完整阅读。

### 兼容性

重点兼容以下场景：

1. Element Plus 的抽屉、表单、表格、滑块、数字输入框在窄屏下的宽度和换行行为。
2. 手机浏览器地址栏和软键盘导致的视口高度变化。
3. 长文本与长路径在窄屏下的换行或截断行为。

## 测试策略

### 手工验收重点

1. 登录页在手机宽度下无横向滚动。
2. 顶部导航、底部导航、菜单抽屉可正常联动。
3. 生成页在 “无任务 / 进行中 / 已完成 / 失败” 四种状态下都能正常阅读和操作。
4. 参数抽屉内的输入框、滑块和按钮不会被遮挡或挤压。
5. 历史页在手机端展示为卡片列表，详情抽屉内容完整。
6. 模型中心与任务审计页的卡片、标签和长文本无溢出。

### 自动化验证范围

本次改造优先保持现有单元测试稳定，并补充必要的组件测试：

1. 公共导航在手机断点下的渲染分支。
2. 历史列表桌面/手机双形态切换。
3. 生成页参数面板在手机端抽屉中的挂载与交互。

## 实施顺序建议

1. 先改 `AppShell.vue`，统一全站手机导航框架。
2. 再改 `GenerateView.vue`、`ParameterPanel.vue`、`ResultPreview.vue`。
3. 再改 `HistoryFilters.vue`、`HistoryTable.vue`、`HistoryDetailDrawer.vue`。
4. 最后统一补齐 `DashboardView.vue`、`ModelCenterView.vue`、`TaskAuditView.vue` 和全局样式收口。

## 验收标准

满足以下条件即可认为本次设计落地成功：

1. 所有主页面在常见手机宽度下无页面级横向滚动。
2. 主导航、次级操作入口和详情入口在手机端均可触达。
3. 生成、筛选、查看详情等关键流程在手机端可完成。
4. 桌面端原有结构和主视觉未出现明显回归。
